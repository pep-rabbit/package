# DeLiki

Monorepo-проект для пошуку аптек за програмою «Доступні ліки» з єдиною точкою входу через Nginx.

Головна ідея: допомогти пацієнту з е-рецептом швидко знайти аптеки, які працюють з НСЗУ і найімовірніше активно відпускають потрібний препарат.

## ТЗ завдання

### «ДеЛіки» - Пошук безоплатних ліків поруч

Пацієнт отримав е-рецепт за програмою «Доступні ліки», але не знає, яка аптека поруч реально відпускає потрібний препарат.

### Датасети data.gov.ua

1. Аптечні заклади за програмою «Доступні ліки» (НСЗУ):
   https://data.gov.ua/dataset/3503ea5a-456d-4780-905b-b74e7d8f09cf
2. Оплати аптекам за реімбурсацією (НСЗУ):
   https://data.gov.ua/dataset/959dca0a-9b74-41ff-a7c8-f8de6398a219

Оновлення: дані НСЗУ оновлюються регулярно протягом 2025 року.

### Функціонал MVP

- користувач вводить місто + назву препарату (МНН);
- система знаходить аптеки з договором НСЗУ;
- ранжує результати за обсягом оплат (більше -> активніше відпускає);
- показує адресу, контакт і тип видачі;
- зберігає «ТОП-5 аптек» у файл.

### Ринкові аналоги

- Табличка НСЗУ
- Helsi.me

## Що входить у проєкт

- `DeLiki/` - backend API (Litestar + Polars), реалізує пошук, фільтрацію та ранжування аптек.
- `Nginx/` - веб-частина: статика (HTML/CSS/JS) + reverse proxy до API.
- `user-cli/` - TUI-клієнт на Textual для ручної перевірки API-сценарію.
- `compose.yml` - контейнерна оркестрація API та Nginx.

## Архітектура

1. Користувач відкриває веб-інтерфейс на `http://localhost/`.
2. Frontend надсилає запит на `/api/top-pharmacies?city=...&medical_program=...`.
3. Nginx проксіює `/api/*` у контейнер `api` (`http://api:8080/api/`).
4. API читає локальні датасети НСЗУ, агрегує виплати, фільтрує по місту/програмі.
5. API повертає відсортований топ аптек, frontend відображає результат і може зберегти JSON-звіт.

## Потік обробки даних

1. `reimbursement_legal_entity_divisions_info.csv` використовується як довідник аптек/відділень.
2. `payments_on_contracts_pharmacy_2025.csv` агрегується по `legal_entity_edrpou` у метрику `activity_score`.
3. Два набори з'єднуються по `legal_entity_edrpou`.
4. Застосовуються фільтри:
    - `division_settlement == city`
    - входження `medical_program` у `medical_programs_in_divisions`
5. Результат сортується за `activity_score` та обрізається до топ-5.

## API контракт

- Endpoint: `GET /api/top-pharmacies`
- Query-параметри:
   - `city`: назва міста
   - `medical_program`: МНН або назва медичної програми
- Поля відповіді:
   - `legal_entity_name`
   - `division_name`
   - `division_addresses`
   - `division_phone`
   - `division_type`
   - `division_settlement`
   - `activity_score`

Приклад:

```text
/api/top-pharmacies?city=Київ&medical_program=Інсуліни
```

## Швидкий старт (Docker Compose)

З кореня проєкту:

```bash
docker compose up --build
```

Після запуску:

- веб-інтерфейс: `http://localhost/`
- API через Nginx: `http://localhost/api/top-pharmacies?city=Київ&medical_program=Інсуліни`

## Локальний запуск без Docker

1. Запустити API (в каталозі `DeLiki`):

```bash
granian src.main:app --host 0.0.0.0 --port 8080 --interface asgi
```

2. Запустити CLI (в каталозі `user-cli`):

```bash
API_BASE_URL=http://127.0.0.1:8080 python main.py
```

## Поточні обмеження MVP

- у веб-інтерфейсі зараз відображаються перші 3 результати, хоча API повертає топ-5;
- пошук медичної програми виконується literal-рядком без розумних синонімів/нормалізації;
- CSV читаються при запитах, без окремого шару кешу/прогріву в пам'яті;
- відсутні авторизація, ролі та персоналізація сценарію користувача.

## Як покращити кожну частину проєкту

### 1) Backend API (`DeLiki`)

- додати валідацію параметрів (`city`, `medical_program`) і нормалізацію вводу;
- додати явну модель помилок (4xx/5xx) і структуровані відповіді для frontend/CLI;
- винести конфігурацію в `.env` (шляхи до CSV, CORS, ліміти);
- додати кешування агрегованих даних (in-memory або Redis) для зменшення латентності;
- покрити логіку юніт- і інтеграційними тестами (endpoint + обробка CSV).

### 2) Web + Nginx (`Nginx`)

- відображати всі топ-5 результатів, щоб UI відповідав бізнес-вимозі MVP;
- додати явні стани інтерфейсу: loading, empty, error, retry;
- реалізувати базову валідацію форми до відправки запиту;
- додати кеш-контроль для статики та стиснення (gzip/brotli) в Nginx;
- додати health-check endpoint і timeout/retry політики для proxy.

### 3) CLI (`user-cli`)

- додати командний режим (аргументи CLI) поряд з інтерактивним TUI;
- реалізувати експорт у файл із налаштуванням формату (`json/csv`);
- додати пагінацію/навігацію по результатах і гарячі клавіші;
- покрити клієнт інтеграційними тестами з моками API.

### 4) Дані та якість результатів

- автоматизувати оновлення датасетів (планувальник + валідація схеми);
- додати контроль якості даних (пропуски, дублікати, невалідні телефони/адреси);
- ввести додаткові сигнали ранжування (свіжість, гео-близькість, режим роботи).

### 5) DevOps та спостережуваність

- додати CI (lint, tests, basic security checks);
- підключити централізоване логування і метрики (latency, error rate);
- додати профілі середовищ (`dev/stage/prod`) та продакшн-конфіг docker compose.
